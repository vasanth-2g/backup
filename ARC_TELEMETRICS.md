# Telemetry Architecture - XGBoost Runtime and RCA Role

This document explains how the telemetry branch is built and how the backend
uses it inside the maintenance RCA flow.

## Purpose

Telemetry is the numeric sensor branch of the app. It estimates:

- remaining useful life in operating cycles
- failure risk
- telemetry severity
- important sensor/features contributing to the risk score

The telemetry result is not the final answer alone. It is fused with vision and
RAG knowledge to produce RCA.

```text
input.txt telemetry history
  -> feature engineering
  -> XGBoost RUL model
  -> XGBoost risk model
  -> telemetry output
  -> RCA with vision + RAG
```

## Training Source

The XGBoost models were built from NASA C-MAPSS turbofan telemetry. The original
training notebooks/data can live outside the app in the backup workspace, but
runtime only needs the saved artifacts copied into the app.

Training concept:

```text
C-MAPSS train/test files
  -> per-engine cycle histories
  -> RUL target creation
  -> rolling sensor features
  -> XGBoost regression/classification
  -> saved model artifacts
```

The model was trained as a tabular time-series baseline after feature
engineering. XGBoost is used because it works well for structured sensor data,
loads quickly, and produces compact runtime artifacts.

## Why C-MAPSS Is Relevant To Robotics

C-MAPSS is not raw real-world flight data. It is generated by a sophisticated
thermodynamic simulator: the Commercial Modular Aero-Propulsion System. The
dataset combines physics-based engine equations with simulated degradation
trajectories, so each unit behaves like a controlled run-to-failure experiment.

That is similar to how robotics teams use digital twins and physics simulators.
In robotics, tools such as Gazebo, MuJoCo, Isaac Sim, or custom digital-twin
models can inject artificial faults into a simulated machine before deployment.
Examples include:

- joint backlash
- increased friction
- encoder or sensor noise
- actuator degradation
- lubrication loss
- bearing wear
- temperature drift

The reason we use C-MAPSS here is the same reason robotics engineers use digital
twins: it gives repeatable sensor histories with known degradation behavior. That
makes it useful for building and testing anomaly detection, remaining-life
prediction, and RCA logic before connecting the system to physical hardware.

In this app, C-MAPSS acts as the telemetry digital-twin substitute while DINOv2
handles the visual inspection branch.

## Saved Runtime Artifacts

Runtime artifacts are stored here:

```text
app/backend/XGboost/telemetry_risk/
```

Required files:

```text
xgb_rul_model.joblib
xgb_risk_model.joblib
feature_columns.json
metadata.json
```

Meaning:

- `xgb_rul_model.joblib`: predicts remaining useful life in cycles.
- `xgb_risk_model.joblib`: predicts failure risk probability.
- `feature_columns.json`: exact feature order used during training.
- `metadata.json`: thresholds and model configuration used at runtime.

## Input Contract

The frontend accepts one scenario file:

```text
input.txt
```

The scenario may contain telemetry as:

```json
{
  "telemetry_history": [
    {
      "unit": 1,
      "cycle": 1,
      "op_setting_1": 0.0,
      "op_setting_2": 0.0,
      "op_setting_3": 100.0,
      "sensor_1": 518.67,
      "sensor_2": 641.82
    }
  ]
}
```

Runtime expects C-MAPSS-style fields:

```text
unit
cycle
op_setting_1 .. op_setting_3
sensor_1 .. sensor_21
```

If telemetry history is missing, the backend marks telemetry mode as
`unavailable` and uses safe fallback values. For real prediction, sensor history
must be present.

## Runtime Backend Flow

Main file:

```text
app/backend/scenario_runtime.py
```

Runtime steps:

```text
POST /api/infer
  -> read input.txt payload
  -> normalize telemetry history
  -> rebuild feature row
  -> load XGBoost artifacts
  -> predict RUL
  -> predict failure risk
  -> calculate severity
  -> calculate feature/sensor attribution
```

Feature reconstruction uses the same idea as training:

- current sensor values
- cycle-to-cycle deltas
- rolling mean over 5, 10, and 20 cycles
- rolling standard deviation over 5, 10, and 20 cycles
- operating settings
- subset indicators if present

The feature row is ordered by `feature_columns.json` before prediction.

## Telemetry Output

Backend output shape:

```json
{
  "predicted_rul": 129.5,
  "failure_risk": 0.0001,
  "severity": "normal",
  "risk_label": 0,
  "evidence": {
    "model_mode": "runtime_xgboost",
    "history_length": 1,
    "last_observed_cycle": 1,
    "top_sensors": []
  }
}
```

Human meaning:

- `predicted_rul`: estimated remaining operating cycles.
- `failure_risk`: probability-like model score for near-term failure.
- `severity`: `normal`, `warning`, or `critical`.
- `top_sensors`: highest contributing sensor groups when attribution is
  available.

Example conversion:

```text
129.5 remaining cycles / 10 cycles per day = about 13 days
```

This conversion is only valid if the operator knows the actual cycles per day.

## Severity Logic

Severity is calculated from RUL and risk thresholds in metadata:

```text
critical -> very low RUL or very high risk
warning  -> low RUL or risk above alert threshold
normal   -> neither warning nor critical
```

Telemetry severity can stay normal even when vision sees oil leak/corrosion.
That means sensors do not yet show strong degradation, while visual inspection
does show a local defect.

## RCA Role

Telemetry is one evidence source for RCA:

```text
Telemetry:
  remaining cycles
  failure risk
  severity
  top sensors

Vision:
  defect type
  defect location
  anomaly score

RAG:
  SOPs
  manuals
  work orders
  safety guidance

LLM/Fallback RCA:
  status
  root cause
  next step
  confidence
```

Typical interpretation:

- telemetry normal + vision anomaly: inspect visual defect, continue monitoring
- telemetry warning + vision anomaly: prioritize maintenance inspection
- telemetry critical: stop or isolate asset before return to service
